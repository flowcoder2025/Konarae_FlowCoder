## 2. 전문가별 상세 분석

### 2.1 📚 크리스텐슨 (Clayton Christensen) - Jobs-to-be-Done

#### 분석 프레임워크
Jobs-to-be-Done 이론에 따르면, 고객은 제품이 아니라 "진전(progress)"을 구매합니다.

#### 현재 IR Deck의 JTBD 전달

| Job 유형 | 현재 전달 | 평가 |
|---------|---------|------|
| 기능적 Job | 적합한 지원사업 찾기, 서류 작성 시간 단축 | ⚠️ 부분적 |
| 감정적 Job | 마감 직전 실수 공포 해소 | ✅ 양호 |
| 사회적 Job | 전문성 인정, 성공적 기업 이미지 | ❌ 미언급 |

#### 핵심 문제점

1. **진짜 Job 명확화 부족**
   - 현재: "자동 매칭"이 핵심 가치로 제시
   - 문제: 고객의 진짜 Job은 "합격"이지 "매칭"이 아님
   - 개선: "합격 확률을 높이는 시스템"으로 재프레이밍

2. **Hiring & Firing 관점 누락**
   ```
   현재 "고용" 중인 대안:
   ├── 컨설턴트 대행 (고비용, 대리작성 리스크)
   ├── 엑셀/노션 자체 관리 (시간 소모, 실수 위험)
   └── 야근 + 운 (비체계적, 재현 불가)

   왜 "해고"하고 Konarae를 "고용"하는가? → 스토리 부족
   ```

3. **Progress Making Forces 불균형**
   - Push (현 상황 고통): ✅ 잘 표현 (선택 마비, 자료 분산, 제출 리스크)
   - Pull (새 솔루션 매력): ⚠️ "자동 점검" 가치 전달 미약
   - Anxiety (변화 불안): ⚠️ "셀프서브 학습" 언급만, 해소 방안 없음
   - Habit (현재 습관): ❌ 다뤄지지 않음

#### 개선 권고

| 슬라이드 | 현재 | 개선 |
|---------|-----|------|
| 1 (타이틀) | "선택부터 검증까지 클로징" | "합격률을 높이는 제출 운영 시스템" |
| 7-8 (Problem) | 문제만 제시 | 이후 "고객의 현재 해결 방식" 슬라이드 추가 |
| 14 (마스터 프로필) | 기능 설명 | "반복 지원 시 70% 시간 절감" 결과 강조 |

### 2.2 📊 포터 (Michael Porter) - Five Forces & 경쟁 우위

#### 분석 프레임워크
Porter의 Five Forces와 가치사슬 분석으로 지속가능한 경쟁우위를 평가합니다.

#### Five Forces 평가

| Force | 현재 분석 | 누락 요소 | 개선 필요 |
|-------|---------|---------|---------|
| 기존 경쟁 | 정부포털, 컨설팅 | SaaS 솔루션, 범용 AI | 🔴 |
| 신규 진입 | 미언급 | AI 스타트업, 빅테크 | 🟡 |
| 대체재 | 컨설턴트만 | ChatGPT, Notion AI | 🔴 |
| 공급자 교섭력 | 미언급 | AI API 의존도 | 🟡 |
| 구매자 교섭력 | 미언급 | B2B vs 스타트업 세그먼트 | 🟢 |

#### 핵심 경쟁 우위 Gap

**현재 경쟁 분석 (슬라이드 9, 28):**
```
정부/포털 vs 컨설팅/대행 vs Konarae
→ "제출 품질 시스템화"를 차별점으로 제시
```

**누락된 경쟁 축:**
```
범용 AI (ChatGPT, Claude, Gemini)
├── 장점: 무료/저렴, 즉시 사용 가능
├── 단점: 공고 실시간 크롤링 불가, 증빙 연결 불가, 정합성 검증 한계
└── Konarae Edge: 도메인 특화 + 데이터 축적 + 시스템 통합
```

#### 가치사슬 분석

| 활동 | 현재 전달 | 가치 포착 명확화 필요 |
|-----|---------|-------------------|
| 데이터 수집 | 크롤링 파이프라인 | 공고 커버리지 수치 |
| 매칭 | 의미론적 검색 | 매칭 정확도 수치 |
| 진단 | 부족항목 분석 | 크래딧 마진 구조 |
| 검증 | 제출 전 점검 | 실수 방지 수치 |

### 2.3 🛡️ 탈레브 (Nassim Taleb) - 리스크 & 안티프래질

#### 분석 프레임워크
Black Swan 이론과 Antifragile 개념으로 리스크 인식과 회복력을 평가합니다.

#### 현재 리스크 다룸
- PRD에는 Risk Assessment 존재
- **IR Deck에는 리스크 슬라이드 없음** ← 주요 문제

#### Black Swan 리스크 (미언급)

| 리스크 유형 | 구체적 시나리오 | 영향도 | 발생 확률 |
|-----------|---------------|-------|---------|
| 정책 리스크 | 지원사업 통폐합, 예산 삭감 | 높음 | 중간 |
| 규제 리스크 | AI 생성 문서 제출 금지 | 높음 | 낮음 |
| 경쟁 리스크 | 네이버/카카오 시장 진입 | 높음 | 중간 |
| 기술 리스크 | AI API 비용 급등/중단 | 중간 | 낮음 |
| 운영 리스크 | 크롤링 대상 접근 차단 | 중간 | 중간 |

#### 안티프래질 요소

**현재 언급된 요소:**
- Data Flywheel (슬라이드 17): 데이터 축적 → lock-in

**추가 필요한 안티프래질 요소:**
```
스트레스 → 더 강해지는 구조:

1. 경쟁 심화 → 더 많은 마케팅 노출 → 인지도 상승
2. 규제 강화 → 컴플라이언스 차별화 → 신뢰도 상승
3. AI 발전 → 기능 고도화 → 품질 향상
```

#### Skin in the Game
- 현재: 창업자 소개가 직책/역할 중심
- 필요: "왜 이 문제를 풀어야 하는지" 개인적 동기/경험

#### 개선 권고

**신규 슬라이드: "리스크 & 완화 전략"**
```
┌─────────────────────────────────────────────────────────────┐
│                    Risk Management                          │
├─────────────────┬─────────────────┬─────────────────────────┤
│ Risk            │ Impact          │ Mitigation              │
├─────────────────┼─────────────────┼─────────────────────────┤
│ 정책 변화       │ 🔴 High         │ 다중 카테고리 확장       │
│ AI API 의존     │ 🟡 Medium       │ 멀티모델 + Fine-tuning  │
│ 대형 경쟁사     │ 🟡 Medium       │ 데이터 해자 + 전문성    │
└─────────────────┴─────────────────┴─────────────────────────┘

💪 Antifragile: 경쟁이 심해질수록 데이터가 쌓여 더 강해지는 구조
```

### 2.4 🕸️ 메도우스 (Donella Meadows) - 시스템 다이내믹스

#### 분석 프레임워크
시스템 사고와 레버리지 포인트 이론으로 비즈니스 구조를 분석합니다.

#### 피드백 루프 분석

**강화 루프 (Reinforcing Loop) - 암시적으로만 존재:**
```
R1: Growth Loop
더 많은 기업 → 더 많은 데이터 → 더 나은 매칭 → 더 높은 만족 → 더 많은 기업

R2: Revenue Loop
더 많은 제출 → 더 많은 크래딧 소비 → 더 많은 수익 → 더 나은 제품 → 더 많은 제출
```

**균형 루프 (Balancing Loop) - 미언급:**
```
B1: Market Saturation
성장 → 시장 포화 → 성장 둔화

B2: Quality vs Scale
빠른 확장 → 품질 저하 → 이탈 증가 → 성장 제한
```

#### 레버리지 포인트 평가

| 레버리지 수준 | 요소 | 현재 전달 | 개선 필요 |
|-------------|-----|---------|---------|
| 12 (패러다임) | "지원사업은 운영 업무" | ✅ 슬라이드 8 | - |
| 11 (시스템 목표) | "제출 완료" vs "합격" | ⚠️ 목표 혼재 | 🔴 |
| 9 (정보 흐름) | 마스터 프로필→문서→증빙 | ✅ 슬라이드 14 | - |
| 3 (시스템 구조) | UX 재설계 | ❌ IR 미반영 | 🔴 |

#### 개선 권고

**플라이휠 시각화 슬라이드:**

           ┌──────────────────┐
           │  기업 데이터 축적  │

                    │

    │                               │
    │         🔄 FLYWHEEL          │

    └───────────────┬───────────────┘

  ┌─────────┬───────┴───────┬─────────┐

### 6.1 수익구조 전환 분석: Free Forever + Credit-only

#### 📚 크리스텐슨 (Jobs-to-be-Done) 분석

**두 가지 Job 분리 전략:**

| Job 유형 | 고객 Job | 지불 의향 (WTP) | 모델 |
|---------|---------|---------------|------|
| **Finding Job** | "나에게 맞는 지원사업 찾기" | 낮음 (정보 탐색) | Free Forever |
| **Success Job** | "이 지원사업에 합격하기" | 높음 (결과 달성) | Credit |

**핵심 인사이트:**
```
고객이 진짜 돈을 지불하는 Job = "합격"
→ 매칭/프로필은 Finding Job (무료로 가치 제공)
→ 진단/생성/검증은 Success Job (Credit 과금)

"Finding"에서 "Success"로 전환될 때 지불 의향 급상승
→ Credit 단가 설정 합리적 근거
```

**전환 포인트:**
- 자동 매칭으로 공고 발견 → "이거 지원해볼까?" (Finding → Success 전환)
- 부족항목 진단 요청 시점 = 구매 결정 시점
- Success Job의 긴급성(마감일)이 WTP를 높임

---

#### 🧭 드러커 (Management Philosophy) 분석

**수익 방정식 검증:**
```
무료 서빙 비용 < Credit 수익 × 전환율
```

**필수 검증 KPI:**

| 지표 | 현재 추정 | 목표 | 검증 방법 |
|-----|---------|------|---------|
| 무료→Credit 전환율 | ? | 5~10% | A/B 테스트 |
| 무료 사용자당 서빙 비용 | ? | < 500원/월 | 인프라 모니터링 |
| Credit 건당 수익 | 1,500~3,000원 | - | 가격 책정 |
| 평균 Credit 구매 빈도 | ? | 3회/월 | 사용 패턴 분석 |

**드러커 원칙 적용:**
```
"기업의 목적은 고객 창조"
→ Free Forever로 최대한 많은 고객 유입
→ Credit으로 "성공을 돕는" 서비스에서 가치 포착

주의: 무료 사용자가 가치 없는 비용 센터가 되면 안 됨
→ 무료 사용자도 데이터 기여 (피드백 루프)
→ 무료 사용자의 네트워크 효과 (추천, 바이럴)
```
